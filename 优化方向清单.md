# RAVEN SQ8 热路径优化方向清单

> 创建时间：2026-06-29
> 基线：QPS=16,729 (ef=50, recall=0.9656, SIFT-1M, 单线程, SQ8 AVX2 + target-cpu=native)
> 原则：一个一个来，每项做到极致再进入下一项，before/after 实验数据驱动

---

## Tier 1：高收益

### OPT-1: f32 `l2_simd` 编译期分发

- **状态**：✅ 已完成
- **问题**：SQ8 距离核已用 `#[cfg(target_feature)]` 消除运行时检测，但 f32 `l2_simd` 每次调用走 `is_avx512_supported()` / `is_avx2_supported()` 运行时检测（atomic load + branch）。rerank（~50 次/查询）+ nav.initialize（~48 次/查询）合计 ~98 次 f32 距离/查询。
- **修复**：与 SQ8 相同的 `#[cfg(target_feature)]` 编译期分发。
- **预估收益**：+2~5% QPS
- **风险**：零（与 SQ8 相同手法，已验证）
- **Before**：median QPS=14,446 (rounds: 13943/14446/15112)
- **After**：median QPS=15,192 (rounds: 16353/14097/15192)
- **Δ**：+5.2% QPS, recall=0.9656 无损

### OPT-2: 部分 rerank（只 rerank top-N 而非全部 ef 候选）

- **状态**：✅ 已完成
- **问题**：`search_sq8` 对全部 ef=50 个候选执行 f32 rerank，但只需 top-10。SQ8 排序与 f32 高度一致（recall=0.9656），top-20 内几乎不可能漏掉 top-10。
- **修复**：只对前 `(k*3).max(30)` 个候选 rerank，candidates 已按 SQ8 距离升序
- **量化**：当前 rerank 占总工作量 ~14%，优化后降至 ~8.4%，节省 ~5.6%
- **预估收益**：+5~8% QPS
- **风险**：需验证 recall 不下降
- **Before**：median QPS=15,192 recall=0.9656
- **After**：median QPS=17,950 recall=0.9636 (rounds: 17950/18162/17941)
- **Δ**：+18.1% QPS, recall -0.2pp（可接受）
- **备注**：N=20 太激进（recall 0.9504, -1.5pp），N=30 (k*3) 为最优 sweet spot

## Tier 2：中收益

### OPT-4: `query_code` 堆分配消除（预分配 encode buffer）

- **状态**：✅ 已完成
- **问题**：`search_sq8` 每次调用 `sq8.params.encode(query)` 返回 `Vec<u8>`（SIFT-128 = 128B 堆分配）。15K QPS 下每秒 15K 次 malloc/free。`visited` 和 `pool` 已预分配复用，唯独 `query_code` 遗漏。
- **修复**：`SQ8Params` 新增 `encode_into(v, buf)` 方法；`GraphSearcher` 新增 `query_code_buf: Vec<u8>` 预分配，搜索时复用
- **预估收益**：+1~3% QPS
- **风险**：零（纯内存复用，编码逻辑不变）
- **Before**：median QPS=17,950 recall=0.9636
- **After**：median QPS=17,856 recall=0.9636 (rounds: 17975/15774/17856)
- **Δ**：-0.5% QPS（噪声范围内），recall 无损
- **备注**：128B 堆分配被 tcache 优化，消除后 QPS 收益不明显。但减少内存碎片，与 visited/pool 零分配模式一致

### OPT-5: `to_sorted_vec()` 堆分配消除

- **状态**：❌ 已否决（负优化）
- **问题**：`greedy_search_sq8` 每次返回 `pool.to_sorted_vec()`，分配 `Vec<(u32, f32)>`（~400B）。16K QPS 下每秒 16K 次分配。
- **修复**：在 `GraphSearcher` 中预分配 `result_buf`，传入搜索函数
- **Before**：median QPS=17,856 recall=0.9636
- **After**：v1 median=15,973 (-10.5%), v2 median=16,959 (-5.0%)
- **Δ**：-5~10% QPS（负优化，已回退，两次跑均确认）
- **原因**：`candidates` 从 `Vec` 变为 `&[(u32, f32)]` 切片引用，rerank 循环中 `*id` 解引用比直接 `id` 多一步，LLVM 优化能力下降。400B 分配被 tcache 优化，消除收益不如代码变换的损失

### OPT-6: `batch_search` SQ8/f32 路径代码去重

- **状态**：⬜ 待实验
- **问题**：`batch_search` 中 SQ8 和 f32 两条路径 ~120 行重复代码
- **修复**：提取公共逻辑为闭包或辅助函数
- **预估收益**：0% QPS，纯代码质量

### OPT-7: Graph prefetch 数量适配 R_max

- **状态**：✅ 已完成（QPS 持平，代码正确性提升）
- **问题**：无条件发 4 条 prefetch（256B），但 R_max=32 → 128B = 2 cache lines
- **修复**：按 `r_max` 计算 cache line 数
- **预估收益**：<1% QPS

## Tier 2.5：新发现优化（2026-06-29 审计）

### OPT-11: `sort_unstable_by` + `f32::total_cmp` 替换热路径排序

- **状态**：✅ 已完成（QPS 中性，代码质量提升）
- **问题**：rerank 排序使用 `sort_by(|a, b| a.1.partial_cmp(&b.1).unwrap_or(Ordering::Equal))`，存在两重开销：
  1. `sort_by` 用 Timsort（稳定排序），对小数组需分配临时 buffer
  2. `partial_cmp().unwrap_or()` 产生 `Option<Ordering>` → 分支预测开销
- **修复**：`sort_unstable_by(|a, b| a.1.total_cmp(&b.1))`
  - `sort_unstable_by`：pdqsort，原地排序，无分配，小数组更快
  - `total_cmp`：Rust 1.62+ 稳定 API，单条 `ucomiss` + `seta/setb`，完全无分支
- **影响范围**：6 处热路径（`search`, `search_sq8`, `search_pq4`, `search_pq8`, `batch_search` SQ8, `batch_search` f32）
- **风险**：零（语义等价，`total_cmp` 对 NaN 也有定义）
- **实验方法**：图缓存消除 ~400s 建图热源，交替 A/B 消除热漂移
  - Baseline v1/v2 median: 15,326 / 14,342（均值 14,834）
  - After v1/v2 median: 14,740 / 14,999（均值 14,870）
- **Δ**：+0.2% QPS（噪声范围内），recall=0.9636 无损
- **备注**：排序 30 个元素绝对耗时 <200ns，无法在 0.7s 的搜索总时间里区分。保留是因为代码质量更好（无 `unwrap_or`、无分配、无分支）

### OPT-12: Rerank 软件流水线预取

- **状态**：❌ 已否决（v1 + v2 两轮均确认负优化）
- **问题**：`search_sq8` rerank 阶段对 top-30 候选逐个计算 f32 精确距离，但 **无预取**。
  - 每个候选的 f32 向量 = 128 × 4B = 512B = 8 cache lines
  - 候选按 SQ8 距离排序（非节点 ID 顺序）→ **随机访问模式**
  - 硬件预取器无法预测随机地址 → 每次访问 ~100ns DRAM 延迟
  - 30 次随机访问 × 100ns = 3μs/查询，17K QPS 下占 ~5%
- **修复 v1**：前瞻预取 `i+1` 号候选的 f32 向量首 cache line（1 条，64B）
- **修复 v2**：前瞻预取 `i+2` 号候选的前 2 条 cache line（128B），`_MM_HINT_T0` → L1
  ```rust
  for i in 0..rerank_n {
      if i + 2 < rerank_n {
          let next_id = candidates[i + 2].0 as usize;
          let ptr = &vectors[next_id * dim] as *const f32 as *const i8;
          unsafe {
              _mm_prefetch::<3>(ptr);       // T0 hint → L1
              _mm_prefetch::<3>(ptr.add(64)); // 第 2 条 cache line
          }
      }
      // ... l2_simd for candidates[i] ...
  }
  ```
- **与 SQ8 搜索的区别**：SQ8 搜索已有 Two-Pass Prefetch（OPT-2），但 rerank 阶段完全裸跑
- **预估收益**：+2~4% QPS
- **风险**：零（预取 hint，不影响正确性）
- **实验 v1**（旧 benchmark, 0.7s/轮, CV ±14%）：与 OPT-13 联合测试
  - Baseline 均值 14,834 vs After 均值 14,870（+0.2%，噪声）
- **实验 v2**（新 benchmark v2, 7s/轮, CV 1.8~3.0%）：与 OPT-13 联合测试
  - Baseline v1/v2: median 14,719 / 14,911（均值 14,815, CV 3.0% / 1.8%）
  - After v1/v2: median 14,770 / 14,492（均值 14,631, CV 4.7% / 2.6%）
  - **Δ = -1.2%**（v2 benchmark CV ~2%，结论可信）
- **否决原因**：
  1. 30 个候选的 rerank 总耗时仅 ~6μs/query，60 条 prefetch 指令本身的开销反而大于 DRAM 延迟收益
  2. `l2_simd` 计算时间（~200ns/向量）已足够让硬件预取器拉入同一向量后续 cache lines
  3. v2 设计改进（lookahead=2, 2 cache lines, T0 hint）仍无法扭转为正收益

### OPT-13: `select_nth_unstable` 部分排序替代全排序

- **状态**：❌ 已否决（v1 + v2 两轮均确认负优化）
- **问题**：rerank 对 30 个候选全排序后取 top-10，但只需前 10 个有序
- **修复**：`results.select_nth_unstable(k - 1)` 分区，再 `results[..k].sort_unstable_by(...)`
  - O(n) + O(k log k) vs O(n log n)：30+33 vs 150 次比较，~2.3x 更少
- **与 OPT-11 叠加**：先 `select_nth_unstable` 再对 `[..k]` 用 `total_cmp` 排序
- **预估收益**：<1% QPS（30 元素排序绝对耗时 <100ns，但叠加免费）
- **风险**：零
- **实验 v1**（旧 benchmark）：与 OPT-12 联合测试，+0.2%（噪声）
- **实验 v2**（新 benchmark v2, CV ~2%）：与 OPT-12 v2 联合测试，**Δ = -1.2%**
  - Baseline 均值 14,815 vs After 均值 14,631
- **否决原因**：30 元素时 `select_nth_unstable` 的 partition 开销（swap + pivot 选择）抵消了减少比较的收益；在 CV ~2% 的新 benchmark 下确认轻微负优化

### OPT-14: BFS 节点重排序（Phase 3.1）

- **状态**：❌ 已否决（数据布局不支持 cache line 共享）
- **问题**：节点 ID 顺序由建图决定（permutation shuffle），图遍历的内存访问模式完全随机
  - HybridBlockedCsr `main_block`：node_id × r_max 访问，ID 不连续 → cache line 不连续
  - SQ8 codes：node_id × dim 访问，同理
  - f32 vectors：node_id × dim 访问，同理
  - 三大数据结构的访问局部性均依赖 ID 顺序，当前 ID 顺序 = 随机
- **修复**：建图完成后，从 medoid 出发做 BFS 遍历，按 BFS 序重新编号节点
  - 图遍历 → 接近顺序的 ID → 邻居列表 / SQ8 码 / f32 向量的 cache line 命中率提升
  - 实现难点：需要同步重排 graph storage + SQ8 codes + f32 vectors + layered nav
- **预估收益**：+3~8% QPS（取决于图结构，Glass/DiskANN 均做此优化）
- **风险**：中（工程量大，需处理所有数据结构的 ID 映射；recall 不受影响）
- **关联**：冲刺计划 Phase 3.1（P2），此前未实施
- **实验**：实现了完整的 `reorder_bfs()` + `LayeredNavigation::remap_ids()`，图缓存交替 A/B
  - Baseline v3/v4: 16,541 / 14,792（均值 15,667）
  - After v2/v3/v4: 15,698 / 15,219 / 15,522（均值 15,480）
  - 交替对矛盾：baseline 16,541→after 15,219 (-8%)，baseline 14,792→after 15,522 (+5%)
- **否决原因**：SIFT-128 维下每个数据结构 per-node ≥128B（SQ8 码 128B=2 lines，f32 向量 512B=8 lines，邻居列表 128B=2 lines）。64B cache line 意味着相邻节点永不共享 cache line，BFS 重排序对 cache line 利用率无实质改善。Glass/DiskANN 受益是因为 packed 布局让多个小向量共享 cache line

### OPT-15: SQ8 无加权距离（图导航专用）

- **状态**：✅ 已完成（raw 成为默认 SQ8 搜索路径）
- **问题**：SQ8 距离核计算 `d = Σ (qa[i] - qb[i])² × scale_sq[i]`，每 32 维需 2 次 `loadu_ps`(scale_sq) + 2 次 `cvtepi32_ps` + 2 次 `fmadd_ps`。但图导航只需 **相对排序**，不需要精确距离
- **修复**：新增 `l2_sq8_raw_avx2` 距离核，跳过 scale_sq 加权：`d_raw = Σ (qa[i] - qb[i])²`
  - i32 累加（跳过 `cvtepi32_ps`），最终水平求和时再转 f32
  - 每 32 维节省 6 条指令（2× cvtepi32_ps + 2× loadu_ps + 2× fmadd_ps → 2× add_epi32）
  - SIFT-128 共 4 chunks，净节省 24 条指令
  - `const RAW: bool` 泛型零分支零代码重复
- **实验**：A/B/A/B 交替 5 轮（100K queries/轮，图缓存）
  - Weighted v1/v2: median 15,450 / 14,364（均值 14,907，recall=0.9636）
  - Raw v1/v2/v3: median 15,315 / 15,929 / 16,455（均值 15,900，recall=0.9680）
- **Δ**：+6.6% QPS, **recall +0.44pp**
- **意外发现**：raw 距离不仅没损失 recall，反而提升了 +0.44pp。scale_sq 加权在扭曲距离排序——高方差维度的 SQ8 量化误差被 scale_sq² 放大，raw 距离反而更接近真实的近邻关系
- **实现**：`search_sq8` 默认使用 raw（`::<true>`），`search_sq8_weighted` 保留旧路径

### OPT-16: f32 `l2_avx512` 双寄存器展开

- **状态**：❌ 已否决（LLVM 自动向量化已做此优化，手动展开 -20% QPS）
- **问题**：f32 `l2_avx512` 每次迭代处理 16 个 f32（1 个 `__m512`），循环 8 次（SIFT-128）
  - 每次迭代的 FMA 依赖前一次的 `sum` → **数据依赖链**，无法利用 ILP
- **修复尝试**：双累加器展开，每次处理 32 个 f32（2 个 `__m512`），两个独立累加器
- **实验**：两轮 benchmark（raw SQ8, ef=50, 100K queries/轮）
  - Baseline (OPT-15 raw): 均值 15,900 QPS
  - Dual-accumulator v1: median 12,439 (-22%)
  - Dual-accumulator v2: median 12,954 (-18%)
- **否决原因**：LLVM `-O3` 自动向量化已对原始单累加器循环做了展开 + 多累加器优化。手动展开干扰了编译器的寄存器分配和指令调度，导致 -20% QPS 回退。教训：在对编译器已优化的简单循环做手动展开前，先 `cargo asm` 验证汇编是否已有 ILP

### OPT-17: `encode_into` 预计算逆尺度

- **状态**：✅ 已完成（代码质量提升，QPS 中性）
- **问题**：`encode_into` 每维度做 `(v[d] - offset[d]) / scale[d]`，除法吞吐 1 cycle（Zen 4）
- **修复**：预计算 `inv_scale = 1.0 / scale` 和 `neg_offset_div = -offset * inv_scale`，用 FMA：`v[d] * inv_scale[d] + neg_offset_div[d]`
- **实验**：median QPS=16,279 recall=0.9680（vs OPT-15 baseline 15,900，+2.4% 但在 CV 4.9% 噪声内）
- **Δ**：QPS 中性（<0.5% 可测范围内），recall 无损
- **备注**：每查询仅 1 次编码（128 维），编码占总时间 <0.3%。保留是因为 FMA 比除法更优雅且与 `l2_sq8_raw` 的设计理念一致（预计算常量，热路径纯算术）
