# RAVEN 优化方案（逐项实验验证）

> **核心原则**：每项优化必须有 A/B 对照实验，用数据说话。
> 有些优化直觉上正确但实际可能起反效果（如降低 r_max 提升 QPS 但 recall 暴跌）。
> 每项优化标注 **风险等级**：🟢 低风险（几乎确定有效）/ 🟡 中风险（需验证）/ 🔴 高风险（可能反效果）。

**基准线**（旗舰实验 2026-06-24，SIFT1M，16 线程）：
- f32: α=1.2, r_max=64, l_build=200, max_iter=2, ef_search=100 → recall@10=0.9961, QPS=2434
- f32: α=1.0, r_max=64, ef_search=50 → recall@10=0.9275, QPS=7611
- ADC+rerank: α=1.2, ef_search=100 → recall=0.9676, QPS=2025

## 第一部分：QPS 优化（查询热路径）

### OPT-1: r_max / ef_search Pareto 扫描 🟡

**假设**：r_max=64 的图度数过高（avg_degree=61.2），每步 greedy_search 需计算 61 次距离。降低 r_max 可显著提升 QPS，但可能牺牲 recall。需要找到最优 Pareto 点。

**风险**：r_max 过低会导致 recall 不可接受。r_max=32 的实验已显示 recall 从 0.996 降到 0.953。

**实验方案**：
```
固定参数：α=1.2, l_build=200, max_iter=2, OPQ+AVQ

变量：r_max ∈ {24, 32, 40, 48, 56, 64}
      ef_search ∈ {50, 100, 200, 400, 800}

数据集：SIFT1M

测量指标：recall@10, QPS, avg_degree, p99_degree
```

**验收标准**：
- 找到 Pareto 前沿上的拐点：recall@10 ≥ 0.99 且 QPS ≥ 4000 的组合
- 如果不存在这样的组合，记录为"r_max=64 是当前架构下的必要代价"

**反效果预警**：如果降低 r_max 后 recall 永远无法回到 0.99+，说明 dense graph 是高 recall 的必要条件，此项优化终止。

### OPT-2: 查询热路径预取策略优化 🟡

**假设**：当前 [vamana.rs](file:///home/juwan/Desktop/RAVEN/src/graph/vamana.rs#L414-L424) 的 `_mm_prefetch` 预取的是 `neighbors[i+1]` 的向量数据。但更有效的策略是预取候选集堆顶节点的**邻居列表**（因为下一步要展开它的邻居）。

**风险**：预取方向错误可能增加 cache pollution，反而降低性能。需要实测。

**实验方案**：
```
方案 A（当前）：预取 neighbors[i+1] 的向量数据
方案 B（新）：预取 candidates 堆顶节点的 neighbors 指针
方案 C（组合）：同时预取 A + B
方案 D（无预取）：删除所有 _mm_prefetch
方案 E（批量预取）：预取 neighbors[i+1] 和 neighbors[i+2] 的向量数据（2-ahead）

数据集：SIFT1M, r_max=64, ef_search=100
测量：QPS（10000 次查询取平均），recall@10（应不变）
```

**验收标准**：方案 B/C/E 的 QPS 优于方案 A ≥ 3%，且 recall 不变。

**反效果预警**：如果方案 B/C/E 的 QPS 低于方案 A，说明预取邻居列表干扰了向量计算的 cache line，回退到方案 A。

**代码位置**：[vamana.rs](file:///home/juwan/Desktop/RAVEN/src/graph/vamana.rs) `greedy_search_vec_reuse` 方法

### OPT-2.5: 距离复用避免重算 — ✅ 已验证

**日期**：2026-06-24

**优化内容**：`greedy_search_vec_reuse` 返回 `Vec<(u32, f32)>` 而非 `Vec<u32>`，`search()` 不再重算候选距离。

**基准线**：SIFT1M, α=1.2, r_max=32, l_build=100, ef_search=100, k=10

| 指标 | 优化前 | 优化后 | 变化 |
|:--|:--|:--|:--|
| recall@10 | 0.9528 | 0.9528 | 不变 |
| QPS | 2454 | 2949 | +20.2% |
| avg_latency | 0.407ms | 0.339ms | -16.7% |

**结论**：距离复用 + software prefetch（方案 A）在 recall 不变前提下 QPS +20.2%。
**决策**：已采纳，commit dc814c8。

### OPT-3: BinaryHeap 替换为更紧凑的数据结构 🟡

**假设**：当前 BinaryHeap 存储 `(OrderedF32, u32)` 元组（8 字节），堆操作涉及大量比较和移动。改用 **pairing heap** 或 **flat sorted vec**（ef_search 通常 < 1000，小规模排序可能比二叉堆更快）。

**风险**：BinaryHeap 是 Rust 标准库高度优化的实现，自制堆可能更慢。flat sorted vec 在插入时是 O(n) 但 cache 友好。

**实验方案**：
```
方案 A（当前）：BinaryHeap<Reverse<(OrderedF32, u32)>>
方案 B：flat sorted Vec（插入时 binary_search + insert，O(n) 但 cache 友好）
方案 C：BinaryHeap<u32> + 独立 distances: Vec<f32>（减少堆元素大小）

数据集：SIFT1M, r_max=64, ef_search=100
测量：QPS, recall@10
```

**验收标准**：方案 B 或 C 的 QPS 优于方案 A ≥ 5%，且 recall 不变。

**反效果预警**：flat sorted vec 在 ef_search=800 时插入开销 O(800) 可能抵消 cache 收益。需要在多个 ef_search 值下测试。

**代码位置**：[vamana.rs](file:///home/juwan/Desktop/RAVEN/src/graph/vamana.rs) `greedy_search_vec_reuse` 方法

### OPT-4: f16 距离计算加速 🟢

**假设**：SIFT1M dim=128，f16 距离计算带宽减半（256 字节 vs 512 字节），在 memory-bound 场景下 QPS 理论翻倍。代码已有 [f16.rs](file:///home/juwan/Desktop/RAVEN/src/distance/f16.rs) 实现，只需接入查询路径。

**风险**：f16 精度损失可能导致 recall 下降。但设计文档说"f16 不改变图结构决策"，只用于查询时的快速 rerank。

**实验方案**：
```
方案 A：f32 全精度查询（当前）
方案 B：f16 距离计算用于 greedy_search，f32 rerank 最终结果

数据集：SIFT1M, r_max=64, ef_search ∈ {50, 100, 200}
测量：recall@10, QPS
```

**验收标准**：方案 B 的 QPS 优于方案 A ≥ 20%，且 recall 下降 < 0.5%。

**反效果预警**：如果 recall 下降 > 1%，说明 f16 精度损失在 greedy_search 的路径选择中造成了不可接受的误差，终止此项。

**代码位置**：需修改 [vamana.rs](file:///home/juwan/Desktop/RAVEN/src/graph/vamana.rs) `greedy_search_vec_reuse` 中的距离计算调用

### OPT-5: 查询时自适应 entry_point 选择 🟢

**假设**：当前 NavigationLayer 的 centroid overlay 在 SIFT1M 上仅提升 QPS 0.5%（navigation_sift1m_verify.log）。但实验用的是 k=1000 个 centroid，可能过多。减少 centroid 数量 + 用更精准的最近 centroid 查找可能有更大收益。

**风险**：centroid 过少可能导致 entry_point 质量下降，recall 受损。

**实验方案**：
```
centroid_count ∈ {10, 50, 100, 500, 1000}（当前 1000）
entry_point 策略：
  A. medoid（当前默认）
  B. 最近 centroid（当前 NavigationLayer 实现）
  C. 最近 centroid 的最近邻居（centroid + 1-hop）

数据集：SIFT1M, r_max=64, ef_search=100
测量：recall@10, QPS, avg_visited
```

**验收标准**：找到 centroid_count 和策略的组合使得 QPS 提升 ≥ 10% 且 recall 不变。

**反效果预警**：如果所有组合的 QPS 提升 < 2%，说明 centroid overlay 在当前架构下收益可忽略，标记为"已验证无显著收益"。

## 第二部分：建图速度优化

### OPT-6: init_random_graph 用 Fisher-Yates 采样替代 HashSet 🟢

**假设**：当前 [vamana.rs](file:///home/juwan/Desktop/RAVEN/src/graph/vamana.rs#L233-L248) 对每个节点用 HashSet 去重采样 r_max 个随机邻居。HashSet 有哈希开销。改用 Fisher-Yates 部分 shuffle 直接从 [0,n) 采样不重复的 r_max 个，O(r_max) 时间 + O(n) 临时空间。

**风险**：Fisher-Yates 需要 O(n) 临时数组（1M × 4B = 4MB），但只分配一次然后复用。几乎不可能有反效果。

**实验方案**：
```
方案 A（当前）：HashSet 去重采样
方案 B：Fisher-Yates 部分 shuffle（预分配 Vec<u32> 复用）

数据集：SIFT1M（1M 节点）
测量：init_random_graph 耗时（秒），建图总耗时
注意：最终图结构会不同（随机性不同），但图质量指标应相当
```

**验收标准**：方案 B 的 init_random_graph 耗时 < 方案 A 的 50%。

**反效果预警**：几乎不可能。最坏情况是性能相当。

**代码位置**：[vamana.rs](file:///home/juwan/Desktop/RAVEN/src/graph/vamana.rs) `init_random_graph` 方法

### OPT-7: medoid 计算用采样替代全量扫描 🟢

**假设**：当前 [vamana.rs](file:///home/juwan/Desktop/RAVEN/src/graph/vamana.rs#L256-L282) `compute_medoid` 遍历全部 1M 向量计算质心再找最近点，O(n×dim)。可以用随机采样 10K 点计算近似质心，O(10K×dim)，快 100 倍。

**风险**：近似质心可能与真实质心不同，导致 entry_point 质量略降。但 medoid 本身只是搜索起点，微小偏差影响极小。

**实验方案**：
```
方案 A（当前）：全量扫描计算 medoid
方案 B：采样 10K 计算近似 medoid
方案 C：采样 1K 计算近似 medoid

数据集：SIFT1M
测量：compute_medoid 耗时，recall@10（应不变），QPS（应不变）
```

**验收标准**：方案 B 的 medoid 计算耗时 < 方案 A 的 5%，且 recall/QPS 不变。

**反效果预警**：如果 recall 下降 > 0.1%，说明 entry_point 偏差影响了搜索质量，回退到全量。

**代码位置**：[vamana.rs](file:///home/juwan/Desktop/RAVEN/src/graph/vamana.rs) `compute_medoid` 方法

### OPT-8: rayon 并行粒度调整 🟡

**假设**：当前 per-node 并行（1M 个 task），rayon 调度开销可能显著。改为 per-batch（每批 1000 节点）可减少调度开销。

**风险**：batch 过大可能导致负载不均衡（某些节点的 greedy_search 耗时远超其他）。batch 过小则调度开销不减。

**实验方案**：
```
batch_size ∈ {1（当前）, 100, 500, 1000, 5000}

数据集：SIFT1M
测量：建图总耗时，recall@10（应不变）
```

**验收标准**：找到 batch_size 使得建图耗时减少 ≥ 10% 且 recall 不变。

**反效果预警**：batch_size 过大可能导致某些线程空闲等待，反而更慢。

## 第三部分：量化层优化

### OPT-9: PQ k-means 初始化改为 k-means++ 🟢

**假设**：[pq.rs](file:///home/juwan/Desktop/RAVEN/src/quant/pq.rs#L140-L141) 的 `kmeans` 函数用"取前 k 个点"初始化，这是最差的初始化策略。改为 k-means++（[avq.rs](file:///home/juwan/Desktop/RAVEN/src/quant/avq.rs) 已实现 `weighted_kmeans_pp`）可提升 codebook 质量。

**风险**：k-means++ 初始化更慢（O(n×k)），但只在训练时执行一次。codebook 质量提升几乎确定。

**实验方案**：
```
方案 A（当前）：取前 k 个点初始化
方案 B：k-means++ 初始化

数据集：SIFT1M learn 100K
测量：
  - PQ reconstruction loss（越低越好）
  - 接入图查询后 recall@10（应不变或提升）
  - PQ 训练耗时（应增加但可接受）
```

**验收标准**：方案 B 的 reconstruction loss < 方案 A，且 PQ 训练耗时增加 < 3 倍。

**反效果预警**：几乎不可能有反效果。最坏情况是 loss 相当。

**代码位置**：[pq.rs](file:///home/juwan/Desktop/RAVEN/src/quant/pq.rs) `kmeans` 函数

### OPT-10: AVQ 归一化方案消融 🟡

**假设**：设计文档修正摘要 #24 要求对比 Mean/StdDev/Mad/LogSumExp 四种归一化方案。代码已实现四种（[quant_aware_prune.rs](file:///home/juwan/Desktop/RAVEN/src/graph/quant_aware_prune.rs)），但只跑了 Mean 的实验。SIFT 数据集密度不均，StdDev 或 Mad 可能更优。

**风险**：不同的归一化方案可能在不同数据集上表现不同，没有"万能方案"。

**实验方案**：
```
归一化方案 ∈ {Mean, StdDev, Mad, LogSumExp}
β ∈ {0.0, 0.1, 0.3, 0.5, 1.0, 2.0}
r_max = 64（旗舰参数）

数据集：SIFT1M
测量：recall@10, 边长分布, 量化误差分布
```

**验收标准**：找到 (归一化方案, β) 组合使得：
1. recall@10 > β=0.0 基线（当前 0.9528，r_max=32）
2. 量化误差分布的均值 < β=0.0 基线
3. 闭合论证链至少 2/3 成立

**反效果预警**：如果所有方案在所有 β 值下 recall 都 ≤ β=0.0 基线，说明量化感知剪枝假设在 SIFT1M 上不成立，需要在其他数据集上验证。

**代码位置**：[quant_aware_prune.rs](file:///home/juwan/Desktop/RAVEN/src/graph/quant_aware_prune.rs) `compute_normalization` 方法

## 第四部分：消融实验补全

### OPT-11: β 假设在 r_max=64 上重新验证 🟡

**假设**：当前消融实验（ablation_experiment.log）在 r_max=32 上 β=0.3 的 recall（0.9487）低于 β=0.0（0.9528），闭合论证链不成立。但旗舰实验显示 r_max=64 的图质量远高于 r_max=32（0.996 vs 0.953）。β 假设可能在高质量图上才生效。

**风险**：建图时间翻倍（β>0 的量化感知建图比标准建图慢 ~40%）。

**实验方案**：
```
r_max = 64, α = 1.2, l_build = 200, max_iter = 2
β ∈ {0.0, 0.05, 0.1, 0.2, 0.3, 0.5, 1.0, 2.0}
ef_search ∈ {50, 100, 200, 400}

数据集：SIFT1M
测量：
  - recall@10（Pareto 曲线）
  - 四层消融指标（边长分布、误差分布、连通度、种子方差）
  - 闭合论证链
```

**验收标准**：存在 β > 0 使得：
1. recall@10 Pareto 曲线在至少一个 ef_search 点上优于 β=0.0
2. 闭合论证链成立

**反效果预警**：如果 β > 0 在所有 ef_search 点上 recall 都 ≤ β=0.0，且误差分布无改善，标记为"β 假设在 SIFT1M 上不成立"。这不是 bug，是研究结果——负面结果也有价值。
