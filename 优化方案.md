# RAVEN 优化方案（逐项实验验证）

> **核心原则**：每项优化必须有 A/B 对照实验，用数据说话。
> 有些优化直觉上正确但实际可能起反效果（如降低 r_max 提升 QPS 但 recall 暴跌）。
> 每项优化标注 **风险等级**：🟢 低风险（几乎确定有效）/ 🟡 中风险（需验证）/ 🔴 高风险（可能反效果）。

**基准线**（旗舰实验 2026-06-24，SIFT1M，16 线程）：
- f32: α=1.2, r_max=64, l_build=200, max_iter=2, ef_search=100 → recall@10=0.9961, QPS=2434
- f32: α=1.0, r_max=64, ef_search=50 → recall@10=0.9275, QPS=7611
- ADC+rerank: α=1.2, ef_search=100 → recall=0.9676, QPS=2025

**验收标准评估**：
- ❌ "recall@10 ≥ 0.99 且 QPS ≥ 4000" 的组合**不存在**。最高 recall≥0.99 且 QPS 最高的是 r_max=48 ef=100 (0.9934/2524)，远低于 4000。
- ✅ 找到 Pareto 前沿拐点：r_max=48 ef=100 是中高 recall（0.993）的性价比最优解。

**关键发现**：
1. **r_max=48 是性价比拐点**：recall=0.9934（接近 0.99），QPS=2524，建图 2831s（比 r_max=64 快 40%）
2. **r_max=64 是超高 recall（≥0.996）的必要代价**：只有 r_max=64 能在 ef=100 达到 recall≥0.996
3. **r_max=24 不实用**：recall 上限 0.9953（需 ef=800），QPS 仅 788
4. **建图时间与 r_max 近似线性**：r_max 每增加 8，建图时间增加约 600-1000s
5. **QPS 与 avg_degree 反相关**：r_max=24 (deg=23.9) QPS=7519 → r_max=64 (deg=61.2) QPS=3234（ef=50）

**结论**：验收标准"recall≥0.99 且 QPS≥4000"在当前架构下不可达，r_max=64 是高 recall 的必要代价。r_max=48 可作为"快速原型/调试"参数（建图快 40%，recall 0.993）。

**决策**：旗舰参数维持 r_max=64（超高 recall 场景）；r_max=48 作为"快速建图"备选参数记录在案。此项优化不修改代码，仅提供参数选择依据。

**反效果预警触发**：降低 r_max 后 recall 在 ef=100 永远无法回到 0.996+（r_max=48 最高 0.9934），证实 dense graph 是高 recall 的必要条件。

### OPT-2: 查询热路径预取策略优化 ✅ 已验证（方案 B，QPS +28%）

**假设**：当前 [vamana.rs](file:///home/juwan/Desktop/RAVEN/src/graph/vamana.rs#L414-L424) 的 `_mm_prefetch` 预取的是 `neighbors[i+1]` 的向量数据。但更有效的策略是预取候选集堆顶节点的**邻居列表**（因为下一步要展开它的邻居）。

**风险**：预取方向错误可能增加 cache pollution，反而降低性能。需要实测。

**实验方案**：
```
方案 A（当前）：预取 neighbors[i+1] 的向量数据
方案 B（新）：预取 candidates 堆顶节点的 neighbors 指针
方案 C（组合）：同时预取 A + B
方案 D（无预取）：删除所有 _mm_prefetch
方案 E（批量预取）：预取 neighbors[i+1] 和 neighbors[i+2] 的向量数据（2-ahead）

数据集：SIFT1M, r_max=64, ef_search=100
测量：QPS（10000 次查询取平均），recall@10（应不变）
```

**验收标准**：方案 B/C/E 的 QPS 优于方案 A ≥ 3%，且 recall 不变。

**反效果预警**：如果方案 B/C/E 的 QPS 低于方案 A，说明预取邻居列表干扰了向量计算的 cache line，回退到方案 A。

**代码位置**：[vamana.rs](file:///home/juwan/Desktop/RAVEN/src/graph/vamana.rs) `greedy_search_vec_reuse` 方法

**实验结果（2026-06-25）**：

微基准 [opt2_bench.rs](file:///home/juwan/Desktop/RAVEN/src/bin/opt2_bench.rs)（SIFT1M base 493MB + 随机图 R=32，10000 查询）：

| 策略 | QPS | avg_latency | 加速比 |
|:--|---:|---:|---:|
| A: 预取 next_vec（当前） | 17,451 | 57.30us | 1.00x |
| B: 预取 heap_neighbors | **22,305** | 44.83us | **1.28x** |
| C: 组合 A+B | 22,082 | 45.29us | 1.27x |
| D: 无预取 | 20,166 | 49.59us | 1.16x |
| E: 2-ahead 预取 | 21,412 | 46.70us | 1.23x |

**关键发现**：
1. **方案 A（当前预取）是负优化**：比无预取（D）还慢 16%！循环内预取 `neighbors[i+1]` 的向量数据，指令开销超过收益
2. **方案 B 最优**：预取堆顶节点的邻居列表，QPS +28%
3. **方案 D（无预取）比方案 A 快**：说明当前预取策略有害，应删除
4. 方案 C（组合）和 E（2-ahead）介于 B 和 D 之间，不如纯方案 B

**siftsmall 验证**（[opt2_verify.rs](file:///home/juwan/Desktop/RAVEN/src/bin/opt2_verify.rs)）：
- recall@10 = 0.9960（>0.95），预取策略改变未影响搜索质量

**接入决策**：已接入 [greedy_search_vec_reuse](file:///home/juwan/Desktop/RAVEN/src/graph/vamana.rs#L423)
- 删除循环内的 `neighbors[i+1]` 向量预取（方案 A）
- 在 pop 后添加 `storage.prefetch_neighbors(top_node)`（方案 B）
- 新增 [HybridBlockedCsr::prefetch_neighbors](file:///home/juwan/Desktop/RAVEN/src/memory/graph.rs#L85) 方法（只发 prefetch hint，不访问数据）

### OPT-3: BinaryHeap 替换为更紧凑的数据结构 ❌ 否决（flat sorted Vec 0.59x）

**假设**：当前 BinaryHeap 存储 `(OrderedF32, u32)` 元组（8 字节），堆操作涉及大量比较和移动。改用 **pairing heap** 或 **flat sorted vec**（ef_search 通常 < 1000，小规模排序可能比二叉堆更快）。

**风险**：BinaryHeap 是 Rust 标准库高度优化的实现，自制堆可能更慢。flat sorted vec 在插入时是 O(n) 但 cache 友好。

**实验方案**：
```
方案 A（当前）：BinaryHeap<Reverse<(OrderedF32, u32)>>
方案 B：flat sorted Vec（插入时 binary_search + insert，O(n) 但 cache 友好）
方案 C：BinaryHeap<u32> + 独立 distances: Vec<f32>（减少堆元素大小）

数据集：SIFT1M, r_max=64, ef_search=100
测量：QPS, recall@10
```

**验收标准**：方案 B 或 C 的 QPS 优于方案 A ≥ 5%，且 recall 不变。

**反效果预警**：flat sorted vec 在 ef_search=800 时插入开销 O(800) 可能抵消 cache 收益。需要在多个 ef_search 值下测试。

**代码位置**：[vamana.rs](file:///home/juwan/Desktop/RAVEN/src/graph/vamana.rs) `greedy_search_vec_reuse` 方法

**实验结果（2026-06-25）**：

微基准 [opt3_bench.rs](file:///home/juwan/Desktop/RAVEN/src/bin/opt3_bench.rs)（SIFT1M base + 随机图 R=32，ef_search=100，10000 查询）：

| 方案 | QPS | avg_latency | 加速比 |
|:--|---:|---:|---:|
| A: BinaryHeap（当前） | 21,543 | 46.42us | 1.00x |
| B: flat sorted Vec | 12,638 | 79.13us | **0.59x** |

**否决原因**：
1. **flat sorted Vec 的 insert 是 O(n) 移动**：每次 push 需要移动约 100-200 个 8 字节元素，即使 cache 友好，移动量也太大
2. **BinaryHeap 的 push 是 O(log n)**：约 7-8 次比较+移动，总移动量远小于 flat vec 的 O(n) memmove
3. **Vec::insert 的 memmove 开销**：即使数据在 cache 中，移动 100-200 个元素的开销比 BinaryHeap 的 7-8 次交换大

**结论**：BinaryHeap 在 ef_search=100 的场景下已是最优选择，flat sorted Vec 不适合。方案 C（BinaryHeap<u32> + distances）实现复杂且预期收益低，不实验。

### OPT-4: f16 距离计算加速 ❌ 否决（dim=128 无收益）

**假设**：SIFT1M dim=128，f16 距离计算带宽减半（256 字节 vs 512 字节），在 memory-bound 场景下 QPS 理论翻倍。代码已有 [f16.rs](file:///home/juwan/Desktop/RAVEN/src/distance/f16.rs) 实现，只需接入查询路径。

**风险**：f16 精度损失可能导致 recall 下降。但设计文档说"f16 不改变图结构决策"，只用于查询时的快速 rerank。

**实验方案**：
```
方案 A：f32 全精度查询（当前）
方案 B：f16 距离计算用于 greedy_search，f32 rerank 最终结果

数据集：SIFT1M, r_max=64, ef_search ∈ {50, 100, 200}
测量：recall@10, QPS
```

**验收标准**：方案 B 的 QPS 优于方案 A ≥ 20%，且 recall 下降 < 0.5%。

**反效果预警**：如果 recall 下降 > 1%，说明 f16 精度损失在 greedy_search 的路径选择中造成了不可接受的误差，终止此项。

**代码位置**：需修改 [vamana.rs](file:///home/juwan/Desktop/RAVEN/src/graph/vamana.rs) `greedy_search_vec_reuse` 中的距离计算调用

**实验结果（2026-06-25）**：

实现了 SIMD f16 距离核（F16C + AVX-512），在 [f16.rs](file:///home/juwan/Desktop/RAVEN/src/distance/f16.rs) 中新增：
- `l2_f16_mixed_avx512`：F16C 将 f16 转 f32 在寄存器中计算，16-wide AVX-512 FMA
- `l2_f16_mixed_avx2`：8-wide AVX2 + F16C 路径
- `l2_f16_mixed_simd`：统一分发（AVX-512 > AVX2 > 标量）
- 4 个单元测试全部通过

微基准 [opt4_bench.rs](file:///home/juwan/Desktop/RAVEN/src/bin/opt4_bench.rs) 结果（SIFT1M base 493MB，10 万次随机访问）：

| 方案 | QPS | avg_latency | 加速比 |
|:--|---:|---:|---:|
| A: f32 AVX-512 (l2_simd) | 11,773,413 | 0.08us | 1.00x |
| B: f16 SIMD (F16C + AVX-512) | 5,930,360 | 0.17us | **0.50x** |
| C: f16 标量 (l2_f16_mixed) | 2,347,103 | 0.43us | 0.20x |

精度：f16 SIMD vs f32 相对误差 = 0.000000（精度无损，因为 SIFT [0,255] 范围在 f16 正规数范围内）

**否决原因**：
1. **F16C 转换开销主导**：f16 SIMD 需要额外的 `_mm256_cvtph_ps` 指令将 f16 转 f32，指令数约为 f32 AVX-512 的 2 倍
2. **dim=128 向量太小**：单次距离计算只有 8 个 AVX-512 chunk（512 字节 f32），F16C 的固定转换开销占比大
3. **带宽收益无法抵消转换开销**：即使数据超出 L3 cache（493MB），带宽减半的理论 2x 收益被转换开销（约 2x 指令）完全抵消

**适用场景**：f16 SIMD 的收益需要更高维度（如 dim=768/1536），让单次距离计算的计算量更大，才能让带宽收益超过 F16C 转换的固定开销。当前 SIFT1M dim=128 不适合 f16 优化。

**决策**：OPT-4 否决，不接入查询路径。SIMD f16 距离核保留在 [f16.rs](file:///home/juwan/Desktop/RAVEN/src/distance/f16.rs) 中，供未来高维数据集使用。

### OPT-5: 查询时自适应 entry_point 选择 — ❌ 已验证无显著收益

**日期**：2026-06-25

**假设**：当前 NavigationLayer 的 centroid overlay 在 SIFT1M 上仅提升 QPS 0.5%（[navigation_sift1m_verify.log](file:///home/juwan/Desktop/RAVEN/navigation_sift1m_verify.log)）。但实验用的是 k=1000 个 centroid，可能过多。减少 centroid 数量 + 用更精准的最近 centroid 查找可能有更大收益。

**已有数据**（SIFT1M, r_max=32, ef_search=100, √N=1000 centroid）：

| 方案 | recall@10 | QPS | avg_latency |
|:--|:--|:--|:--|
| A. medoid entry | 0.9528 | 3098 | 0.323ms |
| B. centroid entry (√N=1000) | 0.9528 | 3114 | 0.321ms |
| 差异 | 0.0000 | +0.5% | -0.6% |

**siftsmall 验证**（[opt5_centroid_ablation.rs](file:///home/juwan/Desktop/RAVEN/src/bin/opt5_centroid_ablation.rs)）：

3 次运行（siftsmall 10K, r_max=32, ef=100, 100 query）：

| centroid_n | Run 1 QPS | Run 2 QPS | Run 3 QPS | recall |
|:--|:--|:--|:--|:--|
| 0 (medoid) | 14481 | 12673 | 14629 | 0.9960 |
| 10 | 10945 (-24%) | 15311 (+21%) | 18099 (+24%) | 0.9960 |
| 50 | 13156 (-9%) | 15475 (+22%) | 17974 (+23%) | 0.9960 |
| 100 | 10558 (-27%) | 13250 (+5%) | 13391 (-9%) | 0.9960 |

**关键发现**：
1. **SIFT1M 上 centroid overlay QPS 仅 +0.5%**（3098→3114），远低于 2% 验收标准
2. **siftsmall QPS 波动极大**（-27% 到 +24%），因为查询集仅 100 个，测量噪声主导
3. **recall 始终不变**（0.9960 / 0.9528），centroid overlay 不影响搜索质量
4. **找最近 centroid 的开销与收益抵消**：centroid entry 减少 greedy_search 迭代，但增加 O(k*dim) 的最近 centroid 查找开销

**否决原因**：
- SIFT1M 上 QPS +0.5% < 2% 验收标准，反效果预警条件满足
- siftsmall 查询集太小，QPS 噪声大，无法验证"减少 centroid 数量有更大收益"的假设
- medoid entry 已足够好（greedy_search 自收敛），centroid overlay 的收益可忽略

**决策**：OPT-5 标记为"已验证无显著收益"。NavigationLayer 代码保留（设计文档要求实现），但默认关闭（`enable_centroid_overlay: false`）。medoid entry 是当前最优的 entry_point 策略。

### OPT-6: init_random_graph 用 Fisher-Yates 采样替代 HashSet — ❌ 否决（导致 recall 回归）

**日期**：2026-06-24（2026-06-25 隔离实验确认否决）

**假设**：当前 [vamana.rs](file:///home/juwan/Desktop/RAVEN/src/graph/vamana.rs#L233-L248) 对每个节点用 HashSet 去重采样 r_max 个随机邻居。HashSet 有哈希开销。改用 Fisher-Yates 部分 shuffle 直接从 [0,n) 采样不重复的 r_max 个，O(r_max) 时间 + O(n) 临时空间。

**微基准数据**（[opt6_bench.rs](file:///home/juwan/Desktop/RAVEN/src/bin/opt6_bench.rs)，n=100K, r_max=64）：

| 指标 | 方案 A (HashSet) | 方案 B (partial_shuffle) | 方案 C (手写 FY) |
|:--|:--|:--|:--|
| 耗时 | 69.57ms | 33.04ms | 36.73ms |
| 加速比 | 1.00x | 2.11x | 1.89x |

**SIFT1M 隔离实验（2026-06-25）**：

| 实验 | OPT-6 | OPT-7 | recall@10 | QPS | 建图时间 |
|:--|:--|:--|:--|:--|:--|
| HEAD（两者都保留） | ✅ | ✅ | 0.3307 | 4178 | 121.6s |
| 只回退 OPT-7 | ✅ | ❌ | 0.3311 | 4940 | 113.8s |
| 只回退 OPT-6 | ❌ | ✅ | **0.9517** | 2885 | 797.3s |
| 两者都回退 | ❌ | ❌ | **0.9517** | 2562 | 852.1s |

**否决原因**：
1. **OPT-6 是 recall 回归的唯一根因**：隔离实验证明 OPT-7（采样 medoid）对 recall 无影响
2. **partial_shuffle 微基准的假象**：微基准只测了采样函数本身的耗时（2.11x 加速），没测对图质量的影响
3. **循环间数组状态污染**：partial_shuffle 改变 indices 数组顺序，后续节点的采样分布被前序节点的 shuffle 结果影响，导致随机图质量严重下降
4. **建图时间异常缩短**：OPT-6 的建图时间仅 121s（vs 正常 852s），说明图结构极差（稀疏/不连通），搜索遍历的节点少所以快，但 recall 也极低

**决策**：OPT-6 否决，恢复 dc814c8 的 HashSet 采样。微基准的 2.11x 加速在实际建图中是负优化（图质量 > 采样速度）。

### OPT-8: rayon 并行粒度调整 ❌ 否决（task 调度开销仅 0.2%，可忽略）

**日期**：2026-06-25

**假设**：当前 per-node 并行（1M 个 task），rayon 调度开销可能显著。改为 per-batch（每批 1000 节点）可减少调度开销。

**理论分析**：
- rayon par_iter 对 Vec 使用二分切分，叶子 task 约 1 个元素
- rayon 线程池固定（16 线程），work-stealing 平衡负载
- task 创建开销约 100ns/task

**实验数据**（[opt8_rayon_batch.rs](file:///home/juwan/Desktop/RAVEN/src/bin/opt8_rayon_batch.rs)，sift_learn 100K）：

| task 调度总开销 | 10ms (100000 × 100ns) |
| 占建图时间比例 | **0.2%** |

**外推 SIFT1M（1M 节点，建图 4754s）**：
- task 调度总开销 = 1M × 100ns = 100ms
- 占建图时间比例 = 100ms / 4754s = **0.002%**

**否决原因**：
- rayon par_iter 的 task 调度开销仅 0.2%（100K）/ 0.002%（1M），远低于 1% 阈值
- par_chunks 反而可能更差：batch 内节点串行执行，无法利用 rayon 的 work-stealing 负载均衡
- 建图瓶颈在 greedy_search + RobustPrune 的计算，不在调度

**决策**：OPT-8 否决，`order.par_iter()` 已是最优并行策略。
