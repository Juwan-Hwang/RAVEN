## 内存带宽分析结论

**判定：COMPUTE-BOUND（不是 memory-bound）**

| Cache 层级 | Working Set | 有效带宽 |
|---|---|---|
| L1 | 6-24 KB | 12.8 GB/s |
| L2 | 48-768 KB | 13.0 GB/s |
| L3 | 1.5-6 MB | 12.8 GB/s |
| RAM | 24 MB | 12.6 GB/s |

**铁证**：L1→RAM 带宽比 = **1.0x**（几乎无下降）。若 memory-bound，应下降 5-6x（288→50 GB/s）。

有效带宽仅占 L1 峰值的 **4.5%**（12.8/288）——内存系统在等 CPU，不是 CPU 在等内存。

**用户原始假设修正**：dim=768→478ns、dim=1536→960ns 的线性关系，不是因为 memory-bound，而是因为标量计算本身就是线性的（每维度 3 FLOPs）。带宽不变证明瓶颈在计算端。

**优化决策**：AVX2 直接有效（8-wide SIMD 可提升 4-8x，内存能跟上）。布局优化和预取不是当前优先级。

pub trait Serializable: Sized {
    fn serialize(&self) -> Vec<u8>;
    fn deserialize(bytes: &[u8]) -> Result<Self, SerializeError>;

    fn save(&self, path: &std::path::Path) -> std::io::Result<()> {
        let bytes = self.serialize();
        std::fs::write(path, bytes)
    }

## 7. 关键结论与缺失项

| 任务 | 可用 API | 状态 |
|------|---------|------|
| 从 `&[f32]` 扁平数据集构建索引 | `VamanaGraph::build(vectors, dim, &config, &mut rng)` | 可用 |
| 从 `Vec<Vec<f32>>` 构建 | **无直接 API**，需先扁平化为 `Vec<f32>`（`vectors[i*dim..(i+1)*dim]`） | 需手动转换 |
| 查询 k 近邻 | `GraphSearcher::new(&vectors, &graph, ef_search).search(query, k)` | 可用 |
| 底层贪心搜索 | `VamanaGraph::greedy_search_vec(...)` | 可用 |
| Beam search | 无独立 API，`GraphSearcher::search` 即 Vamana 风格的带宽度贪心搜索 | 等价实现 |
| 保存索引到磁盘 | `Serializable` trait 已定义，但 **`VamanaGraph` 未实现** | **未实现** |
| 从磁盘加载索引 | 同上 | **未实现** |
| 保存/加载构建元数据 | `BuildMetadata::to_toml()` / `from_toml()` | 仅元数据，非完整索引 |
| 保存/加载距离内核缓存 | `KernelVariant::save/load`（`src/distance/kernel.rs`） | 与图索引无关 |

## 根因：目标与评测不匹配

`retrieval_aware_loss` 优化的是 **inner product 失真**（平行分量）：
```
loss = Σ w * (orig_ip - quant_ip)²
```

但搜索用的是 **L2 距离**，而 L2 和 inner product 的关系是：
```
||x - y||² = ||x||² + ||y||² - 2<x,y>
```

当前 loss 只优化 `<x,y>` 的失真，**完全没有优化 `||x||²` 和 `||y||²` 的失真**（reconstruction loss）。所以：
- inner product 失真下降 42%
- 但 L2 距离失真（MSE）几乎没变（0.137 → 0.137）
- recall 自然也没变

当前实现只优化了三项中的一项。

找到了。`greedy_search_vec` 有一个**致命的逻辑错误**。 
 
 ## 问题所在 
 
 ```rust 
 // 如果结果集已满且当前距离大于结果集最差距离，跳过 
 if results.len() >= l { 
     if let Some(&(worst, _)) = results.peek() { 
         if dist.0 > worst.0 { 
             continue;  // ← 跳过扩展邻居！ 
         } 
     } 
 } 
 ``` 
 
 `results` 是**最大堆**，`peek()` 返回的是距离**最大**的节点。当前节点距离比最差结果还大时，直接 `continue` 跳过了邻居扩展。 
 
 **这是错的。** Vamana/HNSW 的正确逻辑是：即使当前节点不进入结果集，也必须扩展它的邻居，因为邻居可能比它近得多。这个 `continue` 导致图遍历在早期就停止，根本探索不到好的候选。 
 
 ## 修复 
 
 ```rust 
 while let Some(Reverse((dist, node))) = candidates.pop() { 
     // 提前终止：候选集最好的节点都比结果集最差的远，停止 
     if results.len() >= l { 
         if let Some(&(worst, _)) = results.peek() { 
             if dist.0 > worst.0 { 
                 break;  // ← break 而不是 continue，且放在邻居扩展之后 
             } 
         } 
     } 
 
     // 加入结果集 
     results.push((dist, node)); 
     if results.len() > l { 
         results.pop(); 
     } 
 
     // 扩展邻居（无论是否进入结果集都要扩展） 
     for &neighbor in storage.neighbors(node) { 
         if visited.visit(neighbor) { 
             let d = l2_simd( 
                 query, 
                 &vectors[neighbor as usize * dim..(neighbor as usize + 1) * dim], 
             ); 
             candidates.push(Reverse((OrderedF32(d), neighbor))); 
         } 
     } 
 } 
 ``` 
 
 关键两点： 
 1. `continue` → `break`：候选堆是最小堆，一旦堆顶都比结果集最差的远，后面的只会更远，可以终止 
 2. 终止判断移到邻居扩展**之前**，确保当前节点的邻居总是被扩展 
 
 这才是 recall 低的真正原因，和 `neighbor_count` 无关。
