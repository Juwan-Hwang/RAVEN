# 冠军级 ANN 检索系统RAVEN：设计文档 Final（冻结版 · 修订 2）

RAVEN — 名字来源与内涵
缩写含义
R·A·V·E·N 覆盖系统的五条核心技术路线：

字母	含义	对应系统组件
R	RobustPrune	Vamana/DiskANN 风格图构建与 RP-Tuning 后验 α 调优
A	Anisotropic Quantization	AVQ 检索感知量化，优化 retrieval loss 而非 reconstruction loss
V	Vector Navigation	高维向量空间中的图导航与 Pareto frontier 搜索
E	Engine	面向 ann-benchmarks / big-ann-benchmarks 的高性能 Rust 检索引擎

能记住 5000 个以上食物藏匿点的精确位置

在复杂地形中规划最优导航路径，而非贪心走最短步

这三条行为精确映射 RAVEN 的系统目标：高 recall（不遗漏高质量候选）、量化感知（优先保留低误差边）、图导航（在 Pareto frontier 上找最优工作点）。

一句话介绍（README / 论文摘要可直接使用）
RAVEN (Retrieval-Aware Vector Engine with Navigation) is a high-performance Rust library for approximate nearest neighbor search, combining post-hoc α-tuning via RP-Tuning, anisotropic vector quantization, and quantization-aware graph pruning to push the recall–QPS Pareto frontier on ann-benchmarks and big-ann-benchmarks.


> **定位**：把三条有强经验结果支持、但尚未被统一工程化实现的路线合并到一个 Rust 库——RP-Tuning 后验 α 调优、AVQ 检索感知量化、量化误差反向影响图剪枝决策。主战场：ann-benchmarks 单查询 recall–QPS Pareto frontier（fit-in-RAM，单 CPU）；扩展场景：big-ann-benchmarks 100M/1B 级 SSD（主指标：recall@10 at target QPS，按公开协议）；辅助模式：批量查询吞吐评测（不计入主成绩，feature-gate 隔离）。

## 第一层：距离计算核

### 运行时维度分发（避免 binary bloat）

ann-benchmarks 数据集维度在运行时从 HDF5 文件读取，涵盖 25、50、100、128、256、784、960、1536 等多种规格。对所有维度做编译期特化会导致 binary bloat 和编译时间失控。

**推荐实现**：宏预先特化高频维度，其余走动态兜底：

```rust
macro_rules! dispatch_dim {
    ($dim:expr, $kernel:expr) => {
        match $dim {
            64   => $kernel::<64>(),
            128  => $kernel::<128>(),
            256  => $kernel::<256>(),
            384  => $kernel::<384>(),
            768  => $kernel::<768>(),
            960  => $kernel::<960>(),
            1536 => $kernel::<1536>(),
            _    => dispatch_dynamic_slice($dim),
        }
    };
}
```


### 动态兜底路径

`chunks_exact`、固定步长循环、尾部单独处理，有助于 LLVM 识别规则循环并做自动向量化；是否真正生成 FMA 或消除边界检查，以 cargo asm / objdump 和基准结果为准：

```rust
#[inline(always)]
pub fn l2_dynamic(a: &[f32], b: &[f32]) -> f32 {
    assert_eq!(a.len(), b.len());
    let mut sum = 0.0f32;
    let chunks_a = a.chunks_exact(8);
    let chunks_b = b.chunks_exact(8);
    let (rem_a, rem_b) = (chunks_a.remainder(), chunks_b.remainder());
    for (ca, cb) in chunks_a.zip(chunks_b) {
        for i in 0..8 { let d = ca[i] - cb[i]; sum += d * d; }
    }
    for (x, y) in rem_a.iter().zip(rem_b) { let d = x - y; sum += d * d; }
    sum
}
```


### 内核选择策略

三阶段筛选，避免 thermal throttling 误判（AVX-512 在部分 Intel 平台上混合使用时可能因降频导致 wall-clock 时间不如 AVX2）：

```rust
fn select_kernel(dim: usize) -> KernelVariant {
    // 第一阶段：延迟粗筛
    let candidates: Vec<_> = available_kernels()
        .into_iter()
        .filter(|k| measure_latency_ns(k, dim) < LATENCY_THRESHOLD_NS)
        .collect();
    // 第二阶段：瞬时 QPS 筛选
    let finalist = candidates
        .into_iter()
        .max_by_key(|k| measure_qps(k, dim))
        .unwrap();
    // 第三阶段：持续稳定性验证（结果缓存，首次约 5 分钟）
    if !validate_kernel_stability(&finalist, dim) {
        return fallback_kernel(dim);  // 降级并记录原因
    }
    finalist
}

fn validate_kernel_stability(k: &KernelVariant, dim: usize) -> bool {
    let baseline = measure_qps(k, dim);
    let sustained = measure_qps_sustained(k, dim, Duration::from_secs(300));
    sustained >= baseline * 0.95
}
```

支持配置入口：**缓存最优配置**（首次选定后落盘，跳过微基准）、**强制指定内核**（环境变量或配置文件覆盖）。

### 内存对齐策略

GEMM 缓冲区和向量主存储保证 32/64 字节对齐有助于 SIMD 访问效率；aligned vs unaligned load 哪个更快，以具体 CPU 和基准结果为准。

### 精度三层

| 格式 | 用途 | 接入时机 |
| :-- | :-- | :-- |
| `f32` | 构建真值 + 图决策，全程不降精度 | Week 1–2 主线 |
| `f16` | 带宽优化快路径，减少内存传输，不改变图结构决策 | **Week 7+ 按需优化，不阻塞主线里程碑** |
| `PQ/OPQ/AVQ` | 检索感知压缩，优化 retrieval loss | Week 5–6 主线 |

## 第二层：内存布局

### 图结构索引化原则

图冻结后本质上是大块连续数组。用 `u32` 节点 ID 做邻接表、候选集、visited history，而不是到处传 Rust 引用或裸指针：更省空间（4 字节 vs 8 字节）、更易序列化与跨线程传递、符合整个系统的数据导向设计风格。

**节点 ID 边界**：系统支持最大节点数为 2³²−2（约 42.9 亿），`u32::MAX`（`0xFFFFFFFF`）保留为 sentinel 值（标记无效邻居槽位、overflow 链末端等），不分配给实际节点。完全覆盖 big-ann 1B 场景。

### 索引序列化格式（含版本管理）

```
文件头布局（固定 16 字节，little-endian）：
  [0..4]   magic:   0x414E4E58 ("ANNX")
  [4..8]   version: u32，格式变更时递增，当前为 1
  [8..12]  flags:   u32，预留扩展位
  [12..16] crc32:   u32，文件体校验和

加载时：
  版本号不匹配 → 报错并提示重建索引，不静默加载
  校验和不一致 → 报错，拒绝加载
  跨架构加载须做字节序转换
```


### Hybrid Blocked-CSR 图存储

```
主块：每个节点固定 R_max 个槽（64 字节 cache-line 对齐）
      邻居起点 = node_id × block_size，无需 offset 数组
      空槽填充 sentinel = u32::MAX，加载时可快速跳过

overflow 区：度数超 R_max 的极少数节点指向溢出链
      不浪费主存，不影响常规节点性能
```

建图阶段增加**度数分位数统计钩子**：

```rust
fn log_degree_distribution(graph: &VamanaGraph) {
    let overflow_count = graph.nodes().filter(|n| n.degree() > R_MAX).count();
    let p95 = graph.degree_percentile(0.95);
    let p99 = graph.degree_percentile(0.99);
    tracing::info!(overflow_ratio = overflow_count as f64 / graph.len() as f64,
                   p95_degree = p95, p99_degree = p99);
}
```


### Visited 标记（Clear-List 模式，`Vec<u8>`）

```
Generation counter 在 N=10⁸ 时每线程 400MB，32 线程共 12.8GB，直接挤占缓存空间。Clear-List 把重置开销从 O(N) 降为 O(V≤1000)，用 `Vec<u8>` 代替 `Vec<u32>` 再降 4 倍：
```

```rust
struct VisitedTracker {
    visited: Vec<u8>,
    // 按 ef_search × 3 预分配，避免查询热路径上频繁扩容
    history: Vec<u32>,
}

impl VisitedTracker {
    fn new(n: usize, ef_search: usize) -> Self {
        Self {
            visited: vec![0u8; n],
            history: Vec::with_capacity(ef_search * 3),
        }
    }
    fn visit(&mut self, idx: u32) -> bool {
        if self.visited[idx as usize] != 0 { return false; }
        self.visited[idx as usize] = 1;
        self.history.push(idx);
        true
    }
    fn reset(&mut self) {
        for &idx in &self.history { self.visited[idx as usize] = 0; }
        self.history.clear();
    }
}
```

| 方案 | N=10⁸ / 单线程 | N=10⁸ × 32 线程 |
| :-- | :-- | :-- |
| Generation counter `Vec<u32>` | 400 MB | 12.8 GB ❌ |
| Clear-List `Vec<u8>` | 100 MB | 3.2 GB ✅ |

### 批量查询架构（feature-gate 隔离，不污染主路径）

不同查询强行共享 visited 会改变搜索语义，直接损害召回率。批量模式通过 feature-gate 与单查询主路径隔离，使用显式 `QueryContext` 传参，预留 NUMA 亲和性扩展位：

```rust
// #[cfg(feature = "batch_mode")]
struct QueryContext {
    visited: VisitedTracker,
    candidate_buf: Vec<u32>,
    vector_pack_buf: AlignedVec<f32>,
    numa_node: Option<usize>,  // 预留 NUMA 亲和性扩展位，当前不强制绑定
}

fn batch_distance(ctx: &mut QueryContext,
                  queries: &[&[f32]],
                  candidates: &[u32]) -> Vec<Vec<f32>> {
    if candidates.len() <= 8 {
        return scalar_simd_path(queries, candidates);
    }
    pack_vectors_aligned(candidates, &mut ctx.vector_pack_buf);
    gemm_path(queries, &ctx.vector_pack_buf)
}
```

GEMM 阈值（默认 8）进入自动调参空间，以基准结果为准。该模式作为**辅助吞吐评测**，不参与主成绩。

## 第三层：图索引核心

### 主线选择

主线选 Vamana/DiskANN 风格。传统工作流中调整 α 常需重建或重跑构建流程，RP-Tuning 提供了 post-hoc 调优路径，是系统级卖点。

### α 三段式机制

```
① 全局 α（构建时固定，数据集级默认值）
          ↓ 构建完成，无需重建
② RP-Tuning（post-hoc）
   前提：从较高质量基础图出发
   机制：对现有出邻居集合重跑 RobustPrune，蒸馏不同 α 工作点
   约束：不是从稀疏图中凭空恢复已删边；
         若需保留构建时未剪枝的候选集，须明确额外存储成本
         （三种存储方案及退化判定阈值见附录 A）
   → 生成 α=1.0 / 1.2 / 1.5 / 2.0 变体索引
   → 一次构建覆盖整条 Pareto 曲线的多个工作点
          ↓ 可选
③ 局部 α（按区域局部密度动态分配）
   → 高密度区域更大 α 剪短边，低密度区域保留长程边
   → 局部 α 与 β > 0 的交互作为探索性实验，不进入主对比
     （见附录 C）
```


### β 与 α 的交互效应（实验规划约束）

```
第一阶段：固定 β=0（标准 RobustPrune），扫 α baseline
第二阶段：固定最优 α，扫 β
第三阶段：在最优 (α, β) 附近做联合细扫
```


### 量化感知 RobustPrune（核心研究假设）

**归一化打分函数（含数值稳定保护）**：

```
Score = dist / (μ_dist + ε) + β × error / (μ_error + ε)

  dist    = 候选边的几何距离
  error   = 边两端点 AVQ 平行分量量化误差的均值
            error(u,v) = mean(avq_error(u), avq_error(v))
  μ_dist  = 当前节点所有候选邻居的平均几何距离
  μ_error = 当前节点所有候选邻居的平均量化误差
  ε       = 默认 1e-8，防除零导致 NaN/Inf；
            高维/极端数据集下可按维度自适应缩放，作为备选鲁棒性实验项
  β       = 融合权重，约 0.05–2.0 之间有意义，进入调参空间
```

**归一化消融变量**：


| 方案 | 适用场景 | 风险 |
| :-- | :-- | :-- |
| 均值归一化（主方案） | 密度均匀数据集 | 极端密度不均时均值不稳定 |
| 标准差归一化 | 方差差异大时更稳 | 计算量略高 |
| MAD 归一化 | 鲁棒性最强 | 实现复杂度稍高 |
| log-sum-exp / sigmoid | 备选非线性压缩 | 引入额外超参，列为备选实验项 |

### 消融实验设计（论文核心证据）

```
每个 β 值记录四层指标：

1. 平均边长分布（边两端 L2 距离直方图）
   → 证明 β 增大时，长程导航边比例提高

2. 量化误差分布（保留边端点的 AVQ 平行分量误差均值）
   → 证明 β 增大时，图系统性回避量化不稳定节点

3. 图连通度指标（平均出度、最大出度、孤立节点数）
   → 证明量化感知剪枝没有破坏导航连通性

4. 跨随机种子 recall 方差（辅助稳定性指标）
   → 验证量化感知剪枝是否放大构建随机性

闭合论证链：
  高 β → 量化误差低的边比例上升（拓扑证据）
       → recall 提高（性能证据）
       → 图导航路径避开了量化不稳定区域（机制解释）

补充对照组（见附录 E）：
  "先量化后建图"基线——拆分量化本身的收益与图结构感知的收益
```


### 上层导航：双机制

```
默认层：保留随机层级（可与 HNSW 直接对比）
可选层：√N 个 centroid overlay 锚点节点（可关闭）
```

## 第四层：并发构建

### 锁实现策略

优先选择无竞争路径更短的锁实现（如 `parking_lot`），把锁体积、争用延迟和建图吞吐一起放进微基准验证，不提前引用未经实测的具体数字。

### 延迟剪枝策略

```
硬上限 R_max（如 64）：最终图的出度上界
软上限 R_soft = 1.5 × R_max（如 96）：构建期允许临时膨胀

触发时机：
  1. 节点出度超过 R_soft 才触发单节点 RobustPrune
  2. 整个数据集插入完成后，freeze 前做一次全局 final prune

预期效果（待基准验证）：
  RobustPrune 调用次数显著降低，建图速度预期大幅提升
```

实验验证（2026-06-24，siftsmall 10K，α=1.0, r_max=32, r_soft=48）：
```
当前 build 后图状态：
  超过 R_soft 的节点: 0/10000
  超过 R_max 的节点: 0/10000
  avg_degree=18.7, max_degree=32

DelayedPruneController.final_prune 结果：
  图结构完全不变（avg_degree=18.7, max_degree=32 不变）

结论：DelayedPruneController 的 should_prune + final_prune 逻辑
     已由 connect_bidirectional（vamana.rs:507-529）+ VamanaGraph::final_prune
     内联实现，功能完全重叠。
     DelayedPruneController 保留作为诊断工具（prune_count 统计），
     不作为生产构建路径。
```


### 构建可复现性约定

```
构建过程默认使用确定性 RNG：

  算法：ChaCha8（速度快、质量足够、跨平台结果一致）
  种子入口：配置文件或命令行参数显式指定，默认值 42
  并行分片策略：按节点 ID 范围静态划分，不依赖调度顺序

以上三项写入索引元数据随索引文件落盘：
  rng_algorithm: "chacha8"
  rng_seed: 42
  shard_strategy: "static_range"
  build_version: "0.1.0"
```

## 第五层：量化层

```
PQ（基线）
  优化 reconstruction loss，作为对照组

OPQ
  构建前旋转向量使各子空间方差均等，再做 PQ
  开发成本低，是进入 AVQ 的工程桥梁

AVQ（参考 ScaNN score-aware quantization 思路）
  优化 retrieval-aware quantization loss
  核心洞见：量化误差的平行分量对 inner product 高分候选的
            召回率影响远大于正交分量
  → codebook 训练时用检索目标加权

  ⚠️  retrieval-aware 信号来源须在实现阶段明确指定：
      AVQ codebook 训练在图构建之前，不依赖图结构；
      检索感知信号来自 sampled high-score pairs 或预采样近邻对。
      Week 5 同时实现最小双分支，用消融数据决定主线（见附录 F.11）。
      须在论文方法节说明选择理由并讨论合理性与局限性，
      主动讨论局限性而非只论证合理性。

  → 与量化感知 RobustPrune 的归一化误差项直接联动
  实际收益以自家消融实验为准
```

## 第六层：评测自动化

### 微基准工具

- **日常迭代**：divan（轻量、上手快）
- **正式实验与回归**：criterion（统计驱动，适合参数对比和写实验图表）

以下决策须以微基准结果为准：`chunks_exact(8)` 的向量化效果、aligned vs unaligned load、`parking_lot` vs `std::sync`、GEMM 阈值、AVX-512 的端到端 QPS（含持续稳定性）。

### 分阶段接入策略

先用纯 Rust 扁平二进制格式把本地基准和核心内核跑通，再回头接 benchmark 外壳。

### 两种模式

**模式一：ann-benchmarks（主战场）**

```
接入：ann_benchmarks/algorithms/yourlib/
  __init__.py / Dockerfile / config.yml

参数扫描空间：
  M:               [16, 32, 64]
  ef_construction: [100, 200, 400]
  alpha:           [1.0, 1.2, 1.5]
  kernel:          [auto]
  pq_mode:         [pq, opq, avq]
  prefetch_window: [2, 4, 8]
  beta:            [0, 0.1, 0.3, 1.0]
  r_soft_ratio:    [1.3, 1.5, 2.0]
  gemm_threshold:  [4, 8, 16]
```

**模式二：big-ann-benchmarks（扩展目标）**

```
主指标：recall@10 at target QPS（按公开协议，如 T1=10K QPS、T2=1500 QPS）
延迟分位数（p95/p99）：补充观测，不作为主排序指标

100M / 1B 级，必须使用 Clear-List Vec<u8> Visited
内存：压缩向量 + 热节点 + 轻量图摘要
SSD：冷节点全精度向量 + 完整邻接块
布局：按 graph walk locality 联合重排
图遍历逻辑预留异步 I/O 预取接口，当前不实现，接入时不改核心路径
独立评测协议，分阶段接入
```


### 规则驱动 Auto-tuner（含 DAG 冲突校验）

规则层定位：**合法性 / 常识性剪枝（skip），不表达 preference**。所有配置入口（TOML / CLI / 环境变量 / feature flag）必须先合并成唯一最终配置，再统一校验，校验后不允许任何路径静默覆盖：

```rust
fn build_config(toml: &Path, cli: &CliArgs, env: &Env) -> Result<Config, ConflictError> {
    let cfg = Config::merge(toml, cli, env)?;  // 先合并所有来源
    let rules = load_rules("rules.toml")?;
    rules.validate(&cfg)?;                     // 再统一校验，之后 cfg 不可变
    Ok(cfg)
}

rules! {
    skip if avq == true && distance == L2;
    skip if avx512 == true && dim % 16 != 0;
    skip if gemm_path == true && candidate_count <= gemm_threshold;
    skip if batch_mode == true && !cfg!(feature = "batch_mode");
}
```

### 附录 A：RP-Tuning 额外存储方案（Week 5 开始前三选一）

| 方案 | 描述 | 额外存储代价 |
| :-- | :-- | :-- |
| **A（推荐先验证）** | 直接对当前邻域重跑 prune，不保留原始候选集 | 零 |
| **B（妥协方案）** | 每节点存储被剪掉的邻居 ID，re-prune 时候选集 = 当前邻域 ∪ 被剪边 | O(N × (R_soft − R_max)) |
| **C（完整回溯）** | 保留完整构建期候选集，最多 R_soft 条 | O(N × R_soft) |

**退化判定阈值（实验前锁定，不事后 cherry-pick）**：

```
A 方案视为"不退化"，须同时满足：

  指标一（固定 QPS）：
    固定 QPS = 完整重建版本在该数据集上的最高 QPS
    要求：A 方案在该 QPS 下的 recall@10 差距 < 0.5%

  指标二（固定 recall）：
    固定 recall@10 = 完整重建版本的峰值 recall
    要求：A 方案达到该 recall 所需的 QPS 差距 < 3%

  覆盖范围：至少 3 个不同数据集，每个独立判定，不跨数据集平均
  若任一数据集不满足，降级为 B 方案并记录原因
```

### 附录 F：实现细节备忘

#### F.1 序列化约定

```
索引文件默认使用 little-endian 字节序
（匹配主流 x86 服务器，避免跨架构加载时出现静默错误）

文件头 16 字节布局：
  [0..4]   magic:   0x414E4E58 ("ANNX")，little-endian
  [4..8]   version: u32，little-endian，格式变更时递增，当前为 1
  [8..12]  flags:   u32，预留扩展位
  [12..16] crc32:   u32，文件体校验和，little-endian

跨架构加载时须做字节序转换，不静默使用原始字节。
```


#### F.2 bumpalo arena 作用域（已废弃）

```
原设计：bumpalo 用于构建期临时候选集。
实际实现：VisitedTracker 复用已覆盖零分配热路径需求，
         bumpalo 从未被代码使用，依赖已于 2026-06-24 删除。
```


#### F.3 量化误差计算口径

```
打分函数中的 error 项定义：

  error(u, v) = mean(avq_error(u), avq_error(v))

  即边两端点 AVQ 平行分量量化误差的均值，不是只取起点或终点。

实现时须统一此口径，消融实验记录中须标注所用口径版本。
```


#### F.4 RP-Tuning α 扫描范围锁定

```
RP-Tuning 的 α 测试范围在实验开始前固定：

  扫描范围：α ∈ [0.8, 3.0]
  离散取点：[0.8, 1.0, 1.2, 1.5, 2.0, 3.0]
  边界测试：
    α < 基础图构建 α 时是否完全退化
    α > 2.0 时是否出现 recall 拐点

范围在实验前锁定，不根据初步结果回调上下界。
若初步实验显示范围明显偏移，须在实验记录中注明并重新锁定，
不做静默调整。
```


#### F.5 SSD 扩展接口方向

```
big-ann 路径预留异步 I/O 预取接口，而非 CPU 软件预取指令。
原因：big-ann SSD 赛道核心瓶颈是 I/O 吞吐，软件预取只能掩盖
     L2/L3 延迟，无法覆盖 SSD 访问延迟。

当前主线不实现，仅预留接口签名，接入 big-ann 时不改核心路径。
```


#### F.6 Cache-line 竞争可选优化

```
若后续 profiling 显示存在 cache-line 竞争，优先检查：
  1. 查询统计计数器（共享 atomic 计数，容易伪共享）
  2. QueryContext 头部字段（线程本地状态元数据）
  3. 并发构建时的节点锁数组（若多锁紧密排列）

处理方式：增加 cache padding 或 #[repr(align(64))]。

visited 数组本体通常不是伪共享热点，真正的热点往往是
小型共享元数据而非大块 thread-local 数组。
```


#### F.7 构建可复现性约定

```
构建过程中所有随机行为默认使用确定性 RNG：

  算法：ChaCha8（速度快、质量足够、跨平台结果一致）
  种子入口：配置文件或命令行参数显式指定，默认值 42
  并行分片策略：按节点 ID 范围静态划分，不依赖调度顺序

以上三项写入索引元数据随索引文件落盘：
  rng_algorithm: "chacha8"
  rng_seed: 42
  shard_strategy: "static_range"
  build_version: "0.1.0"

论文复现实验时直接读取元数据，确保不同机器、不同编译版本下
相同参数构建的图结构完全一致。
```


#### F.8 内积（IP）距离适配边界

```
当前文档核心围绕 L2 距离展开，规则层明确标注互斥保护：
  skip if avq == true && distance == L2

若后续扩展支持内积（IP）检索，须注意：

  1. 打分函数归一化基准需单独校准
     IP 距离的数值范围与 L2 不同，μ_dist 量级也不同；
     直接复用 L2 下的 β 默认值会导致权重严重偏移

  2. AVQ 量化误差定义在 IP 场景下有所不同
     L2 场景下的"平行分量误差"需对应调整为 IP 场景的误差分解

  3. β 的有效范围需重新实验确定
     L2 下约 0.05–2.0 有意义的范围不能直接沿用

结论：IP 场景下打分归一化口径与 β 范围需单独校准，
      实现时用独立的 IpQuantAwarePrune 替代 L2QuantAwarePrune，
      不共用参数。
```


#### F.9 内存占用速算公式

```
单次部署的总内存估算（fit-in-RAM 场景）：

  总内存 ≈ 向量区 + 图主块 + overflow 区 + 查询上下文 + 元数据

各项展开：

  向量区     = N × dim × sizeof(dtype)
               例：1M × 768 × 4B (f32) = 3.07 GB
                   1M × 768 × 1B (PQ)  = 0.77 GB

  图主块     = N × R_max × 4B (u32 节点 ID)
               例：1M × 64 × 4B = 0.26 GB

  overflow 区 = overflow_ratio × N × R_soft × 4B
               overflow_ratio 由度数分位数钩子实测（预期 < 1%）

  查询上下文 = T × (N × 1B visited + ef_search × 3 × 4B history)
               例：32线程 × (1M×1B + 600×4B) ≈ 32.1 MB

  元数据     = 忽略不计（KB 级）

快速估算示例：
  N=1M, dim=768, f32, R_max=64, 32线程
  ≈ 3.07 + 0.26 + ~0 + 0.03 ≈ 3.36 GB  ✅ fit-in-RAM

  同上，换用 AVQ (1B/vec)
  向量区降至 0.77 GB → 总计 ≈ 1.06 GB  ✅ 余量更大

big-ann 场景（N=100M）：
  仅图主块 = 100M × 64 × 4B = 25.6 GB，超过普通 RAM
  → 必须启用压缩向量 + SSD offload，解释了 big-ann 需要单独路径的原因
```


#### F.10 AVX-512 持续稳定性测试约定

```
三阶段内核选择后，对进入决赛的候选增加稳定性验证：

  测试方式：连续运行 5 分钟，每 30 秒采样一次 QPS
  通过标准：steady-state QPS ≥ 瞬时 QPS × 0.95
  失败处理：降级到下一候选，记录降级原因和平台信息

实现成本：结果缓存后不重复执行，首次约 5 分钟。
目的：区分瞬时 microbenchmark 和真实 steady-state 吞吐，
      避免 thermal throttling 导致线上表现与测试不符。
```

#### F.13 死代码清理实验结论（2026-06-24）

```
背景：项目进入打榜阶段，全面扫描 dead_code 警告，对每个死代码项
     "先实验验证是否有用，确认没用才删除"。

核心库（src/ 非 bin/）：零死代码，全部被生产路径或测试调用。

实验脚本（src/bin/）死代码清理：

1. avq_recall_probe.rs（Week 6 早期验证脚本）：
   - quantized_recall / adc_recall / adc_recall_rerank 三个函数
     定义了但 main 只调用 f32_recall。
   - 修复：main 现调用全部四个实验，siftsmall 验证结果：
       f32 baseline:    recall@10=1.0000
       quantized (错):  recall@10=0.7300  ← 用量化向量建图，违反 ADC 原则
       adc (正确):      recall@10=0.7300  ← f32 建图 + 量化距离
       adc+rerank:      recall@10=1.0000  ← 两阶段，达标
   - 结论：三个函数都有实验价值（对比 ADC 原则 + rerank 增益），
           保留并接入 main，死代码警告消除。

2. sift1m_bench.rs：
   - recall_at_k 工具函数定义了但 main 内联了 recall 计算。
   - 修复：main 三处内联逻辑（f32/ADC/ADC+rerank）改用 recall_at_k，
           消除代码重复。数学等价（hits/(nq*k) == sum(recall_at_k)/nq）。

3. beta_scan.rs：
   - adc_recall_rerank 函数定义了但 main 内联了含预热+计时的 rerank 逻辑。
   - 结论：函数版本无法满足 main 的预热+计时需求，且会重复创建 quantized_db。
           删除函数，main 内联版本更高效。beta_scan 验证结果不变
           （β=0.5 recall=0.7330/0.9920, β=1.0 recall=0.7300/0.9880）。

4. bandwidth_probe.rs：
   - BandwidthResult.elapsed_us 字段从未读取（main 只打印 latency_ns）。
   - 修复：删除冗余字段（latency_ns 已足够，elapsed_us = latency_ns * n_pairs / 1000）。

5. appendix_a_experiment.rs：
   - ExperimentResult.method 和 build_time_s 字段从未读取。
   - 修复：退化判定中打印 method + build_time_s 对比，让字段被读取，
           提供更完整的诊断信息（rebuild=Xs vs rp_tuning_A=Ys）。

6. kernel_select.rs：
   - verbose 变量声明并赋值但从未读取（未完成的 verbose 模式）。
   - 修复：删除 verbose 变量和 --verbose 参数。

最终状态：cargo build --all-features 零 "never used" / "never read" 警告。
```
